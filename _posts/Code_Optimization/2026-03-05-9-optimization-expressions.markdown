# 최적화 하기 위한 표현들

### **※ 비용이 낮은 연산자가 있다.**

### **※ short circuit 원리**

```c
void f1()
{
		int x = 10, y = 20;
		if((x < 5) && (y = 10))
				x += 10;
		printf("x=%d, y=%d\n", x, y);
}
```

"if((x < 5) && (y = 10))" 

원래 의도는 "y==10"이었을텐데, 문법 오류가 아니라 컴파일러는 그냥 넘어가고 코드대로 실행될 것이다.

if문에서 두 개의 비교를 하고 이 두 개의 비교가 참이어야 실행문이 수행된다.

y = 10은 참일까? ⇒ 참이다 (0은 거짓을, 0이아닌 모든 값은 참으로 인식)

y = 10은 우선 y에 10을 대입한다 → 이것은 0이아닌 값이기 때문에 참으로 인식

그러나 결과는

```c
x = 10, y = 20
```

이렇게 출력 된다.

왜냐면 if문은 x<5가 거짓이고 &&는 모두 참이어야 전체가 참이므로 뒤의 비교문을 수행하지 않더라도 거짓은 자명한 사실이다.

이럴 때는 뒤의 불필요한 비교문은 아예 수행하지 않아 시간을 단축한다. 

논리합과 논리곱에서 첫 요소를 통해 참이나 거짓이 결정될 때 뒤의 비교문을 수행하지 않는 것을 short circuit 원리라고 한다. 

&& : 논리곱인 경우에 첫 비교문이 거짓이면 모두 거짓이므로 그 뒤의 비교문은 모두 수행하지 않는다

(논리곱은 거짓일 확률이 높은 문장을 앞쪽에 둬서 최적화 함) 

|| : 논리합의 경우에는 첫 비교문이 참이면모두 참이 되므로 그 뒤의 비교문은 모두 수행하지 않는다.  

(논리합은 참일 확률이 높은 문장을 앞쪽에 둬서 최적화 함)

### **※ 연관된 표현은 묶어서 처리**

연관된 연산은 하나로 묶어서 처리하는 것이 명령어 수를 줄여 속도를 빠르게 한다.

```c
예 1)
a = b / c + d / c;

예 2)
if((a>=0 && a<10) && (b>=0 && b<10))
{
		f1();
}

예 3)
if((x==1) || (x==2) || (x==4) || (x==8))
}
		f1();
}

예 4)
int test(int a, int b, int c, int d,
          int e)
{
		if(a>0)
			if(b>0)
				if(c==0)
					if(d<0)
						if(e>0)
							return 1;
		else
			return -1;
}

예 5)
z = x % 4;
switch(z)
{
		case 3:
				x += 1;
				break;
		case 2:
				x += 2;
				break;
		case 1
				x += 3;
}
```

```c
예 1)
a = (b + d) / c;

예 2)
if((unsigned)(a|b) < 10)
{
		f1();
}

예 3)
if(x & (x-1) == 0 && x != 0)
{
		f1();
}

예 4)
int test(int a, int b, int c, int d,
          int e)
{
		if(a>0 && b>0 && c==0 && d<0 && e>0)
				return 1;
		else 
				return -1;
}

예 5)
x += 3;
x &= ~3;
```

예 2)

a,b 모두 0~9 사이의 정수 값을 가져야 참

0 이상이라는 것은 양수를 의미하고, 10보다 작은 범위를 나타내므로 (unsigned)(a|b) <10 과 같이 표현 

예5)

x 값을 4의 배수로 만들기 위한 코드로써 

왼쪽 소스는 4로 나눈 나머지에 따라 4의 배수가 되는데 필요한 값을 더해주는 것 이고,

오른쪽 소스는 x에 무조건 0011(2진수) == 3을 더해주고 x의 끝 두자리를0으로 만들어 4의 배수로 만들고 있다 (걍 이해하고 받아들이기) 

왼쪽보다 오른쪽이 쬐금 속도면에서 빠르지만 가독성은 왼쪽이 더 좋다.

### **※ 컴파일러에서 하는 최적화**

c 소스에 대한 어셈블리 코드 만드는 명령

gcc -c -S hello.c

c코드와 어셈블리 코드 함께 볼 수 있는 objdump 명령

gcc -c -g -o hello.o hello.c

objdump -d -S hello.o

 

1. 상수 대체

상수는 컴파일 시 계산되므로 프로그램 변수에서 상수를 사용한다면 최적화를 이용하여 런타임이 아닌 컴파일 시 미리 계산해서 사용할 수 있다.

```c
void main()
{
		int a,b,c;
		
		a = 10;
		b = a + 20;
		c = b + 30;
		
		printf("%d", c);
}
```

위 코드에서 변수 a,b,c는 컴파일 시 값이 결정된다.

a가 10이기 때문에 b는 30, c는 60으로 예측 가능하다. 

만일 최적화를 사용하지 않으면 런타임에 변수 값을 읽어와 연산을 수행하지만, -O2 최적화를 사용하면 상수를컴파일 시에 미리 계산해서 사용하므로 그만큼 시간을 절약 할 수 있다.

  

1. 중복 표현 제거

-O2 옵션 이용하면 중복되는 표현을 생략한다. 

```c
void main()
{
		int a,b,c;
		
		scanf("%d %d", &a, &b);
		
		c = a + b;
		printf("%d", c);
		if((a+b) > 10)
				c = (a+b)*2;
		else
				c = (a+b)*5;
}

<최적화 안할때>
gcc -c -g -O0 -o test.o test.c
objdump -d -S test.o

<최적화 할때>
gcc -c -g -O2 -o test.o test.c
objdump -d -S test.o
```

위에 보면 a+b가 여러 곳에 나타나고 있는데, 이 소스를 최적화 옵션 사용 안하고 컴파일하면 a+b를 매번 계산하는데, -O2 옵션 사용하면 최적화해서 반복 수식 계산이 생략된다.  

1. 데드 코드 제거

호출되지 않는 함수, 도달 할 수 없는 조건이나 코드같은 데드 코드는 컴파일러 입장에서 실행 확률이 없으므로 컴파일러 최적화를 적용시켰을 때 컴파일러가 지맘대로 이런 코드 제거한다. 

```c
void main()
{
		int a, b=0;
		scanf("%d", &a);
		if(a<0 && a>0)   // 도달할 수 없는 조건, 데드 코드임
				 b = 10;
				 
}

<최적화 안할때>
gcc -c -g -O0 -o dead.o dead.c
objdump -d -S dead.o

<최적화 할때>
gcc -c -g -O2 -o dead.o dead.c
objdump -d -S dead.o
```

의도적으로 만든 데드 코드의경우 volatile 사용해서 컴파일러가 멋대로 제거하지 못하게 막아야 한다. 

1. 연산 강도 경감

 비용이 비싼 연산을 비용이 좀 덜한 표현으로 바꿔주는 방법으로 반복문에서 사용하면 효과가 크다 

컴파일러가 알아서 바꿔줌 옵션에 따라서  

```c
void main()
{
		int i, j;
		for(i=0; i<10; i++)
		{
				j = i *5;
				printf("%d", j);
		}
}

<최적화 안할때>
gcc -c -g -O0 -o red.o red.c
objdump -d -S red.o

<최적화 할때>
gcc -c -g -O2 -o red.o red.c
objdump -d -S red.o
```

"j = i *5;" 최적화 전에는 원래 수식대로 * 연산을 취하지만 -O2옵션을 통해 최적화를 하게 되면 * 보다 비용이 싼 add 명령을 이용해 5씩 증가시킨다. 

(어셈블리 확인) 

### **※ APCS (ARM Procedure Call Standard)**

함수를 호출하는 표준

인자, 리턴값이 저장되는 위치, 함수 호출 시 발생하는 여러동작 이해, 이걸 최적화에 적용 가능

레지스터 랑 인자 전달 관계

0~3

4~9 : 서브루틴의 지역변수 저장, 포인터에 의해 접근되는 변수는 레지스터에 저장될 수 없음

10~15 : 특수 레지스터 그룸

서브루틴호출 (함수 호출)

복귀

결과값리턴 

ARM 레지스터활용팁
